# Life-day protocol

이 문서가 기상·취침·생활일 전환 절차의 유일한 정본이다.

`day todo`는 특정 생활일의 일이고, `pending`은 잊으면 안 되는 의무나 후속 작업이다. `someday`는 날짜 없이 선택적으로 보관하는 후보이며 상세 규칙은 [`protocols/someday.md`](someday.md)에 있다.

## 생활일 정의

- 생활일은 `기상` 보고로 시작한다.
- 다음 `기상` 전까지 같은 생활일이다.
- 자정은 경계가 아니다.
- 파일명은 `wake_at`의 `Asia/Seoul` 달력 날짜다.
- 같은 달력 날짜에는 생활일 파일을 하나만 만든다.

## 기상

1. 사용자가 말한 실제 기상 시각을 확인한다.
   - 시각을 말하지 않으면 메시지 수신 시각을 사용한다.
   - "두 시간 전에 일어났어"처럼 말하면 소급 시각을 사용한다.
2. Calendar를 조회하고 `data/indexes/calendar.json`을 갱신한다.
   - `neoctl calendar refresh` 실행 (Google API 직접 호출 → import까지 자동)
3. `neoctl day wake --at <ISO_DATETIME>`를 실행한다.
   - 기존 활성 생활일은 새 기상 시각에 닫힌다.
   - 이전 생활일의 `sleep_at`이 없더라도 추측해서 채우지 않는다.
   - 새 생활일의 medications는 빈 이벤트 배열로 시작하며 기본 약 목록은 없다.
4. medical 상태를 확인하고 알려준다.
   - `neoctl medical list`로 반복 병원 일정 상태를 확인한다.
   - D+N, 마지막 주사일, 다음 주사 예정일, 오버 일수를 알려준다.
   - 오버 3일 이상이면 ⚠️ 표시와 함께 상기시킨다.
   - 다음 주사까지 3일 이내면 ⏰ 표시와 함께 알려준다.
   - 가장 가까운 병원 일정과 남은 날짜를 직접 말한다.
5. 미완료 pending을 확인하고 알려준다.
   - `neoctl pending list` 또는 `brief.md`에서 미완료 항목 조회.
   - 있으면 "밀린 일이야: ..." 하고 보여준다.
6. 지난 생활일의 meals, outings, mood가 미기록 상태면 물어본다.
   - 전날 식사 기록이 비어 있으면 "어제 뭐 먹었어?" 하고 물어본다.
   - 전날 외출 중 안 돌아온 것으로 되어 있으면 확인한다.
   - 전날 기분이 기록되지 않았으면 가볍게 물어본다.
7. 작업에 영향을 줄 외부 일정을 자연어로 요약한다.
8. 오늘 프로젝트 작업을 어떤 형태로 둘지 묻는다.
   - 프로젝트 작업을 주된 일정으로 삼을 수 있는지
   - 외부 일정과 병행할 수 있는지
   - 오늘은 프로젝트 작업을 계획하지 않을지
9. 답을 받은 뒤 `neoctl day capacity 1|0.5|0`을 실행한다.
10. **첫 끼 식사가 기록된 뒤에 선제적으로 묻는다.**
    - "오늘 하고 싶은 일 있어?"
    - 사용자가 말한 것을 todolist나 pending으로 기록할지 판단한다.
      - 오늘 안에 끝낼 수 있는 것 → `neoctl day todo add`
      - day-independent하게 지속되는 것 → `neoctl pending add`
11. 활성 프로젝트, Calendar 날짜, 남은 작업량, 열린 태스크를 확인한다.
12. 오늘 할 태스크를 최대 두 개 후보로 제시하고 추천은 하나로 한다.
13. 사용자가 선택하면 `neoctl day work-plan-task`를 실행한다.
    - Calendar에 예정 작업 일정을 만들었으면 `--calendar-event-id`로 연결한다.
14. 외부 일정 snapshot이 필요하면 `neoctl day snapshot-calendar`를 실행한다.
15. `brief.md`가 자동 재생성됐는지 확인한다.

## 하루 중 — 일상 기록

사용자가 말한 내용을 수동적으로 기다리지 않고, 다음 상황에서 선제적으로 기록을 제안하거나 물어본다.

### 식사 감지

사용자가 무언가 먹었다고 말하면 tag를 추론해서 기록한다.

- "라면 먹었어" → "첫 끼야?" 하고 물어본 뒤 `neoctl day meal add --tag lunch --what "라면"` 실행
- tag를 모르면 끼니 순서대로 자동 배정한다: 첫 끼=아침, 둘째 끼=점심, 셋째 끼=저녁. 간식은 별도.
- 시각을 말하지 않으면 메시지 수신 시각을 사용한다.
- 오후 3시가 지났는데 meals가 비어 있으면 "오늘 아직 아무것도 안 먹은 거야?" 하고 물어본다.
- **선언과 완료를 구분한다.**
  - "밥 올려놨다", "라면 끓이는 중" → 선언 시각에 기록, what에 "(준비 중)" 표기
  - "다 먹었어", "배불러" → 완료 시각에 기록 업데이트
  - 선언만 있고 완료 확인이 없으면 나중에 "밥 다 먹었어?" 하고 확인한다.

### 외출 감지

사용자가 외출을 언급하면 기록한다.

- "다이소 갔다 왔어" → `neoctl day outing go --place "다이소"` 후 바로 `neoctl day outing back`
- "다이소 가는 중이야" → `neoctl day outing go --place "다이소"`만 실행
- 돌아왔다는 말이 없는데 한참 지났으면 "돌아왔어?" 하고 물어본다.
- 용도를 말하면 `--purpose`도 기록한다.

### 기분 감지

사용자가 기분을 표현하면 기록한다.

- "오늘 좀 피곤하네" → `neoctl day mood set --summary "피곤함"`
- 이유를 말하면 `--reason`도 기록한다.
- 저녁 무렵까지 mood 기록이 없으면 "오늘 기분 어땠어?" 하고 가볍게 물어본다.

### 샤워/목욕 감지

사용자가 샤워나 목욕을 언급하면 기록한다.

- "씻고 올게" → `neoctl day shower go --type shower`만 실행
- "씻었어", "씻고 왔어" → `neoctl day shower back SHOWER_ID --at <시각>`
- "목욕할래" → `neoctl day shower go --type bath`
- 시각을 말하지 않으면 메시지 수신 시각을 사용한다.
- 시작 없이 완료만 보고하면, 머리 말리기 포함 약 1시간 전으로 소급 시작 시각을 제안한다.
- 완료 없이 시작만 하고 한참 지났으면 "다 씻었어?" 하고 확인한다.

### 머리 말리기 감지

샤워 후 머리 말리기를 언급하면 기록한다.

- "머리 말릴게" → `neoctl day hair-dry go`만 실행
- "머리 다 말렸어" → `neoctl day hair-dry back HAIR_DRY_ID --at <시각>`
- 시각을 말하지 않으면 메시지 수신 시각을 사용한다.
- 시작 없이 완료만 보고하면, 샤워 종료 시각부터 소급 시작 시각을 제안한다.
- 완료 없이 시작만 하고 한참 지났으면 "머리 다 말렸어?" 하고 확인한다.

### 할 일 감지

- "이거 해야 하는데..." → todolist나 pending으로 기록 제안
- "다음에 해야지" → pending으로 기록

## 약 기록 조회

- 약의 마지막 복용이나 복용 여부를 조회할 때는 현재 생활일만 보지 말고 `neoctl day med history`로 모든 생활일을 읽는다.
- `--name`은 양끝 공백을 무시하고 대소문자를 구분하지 않으며, 필요하면 `--action taken|skipped`, `--since`, `--until`, `--limit`을 사용한다.
- 사용자에게 언제 복용·도포했는지 말할 때는 `occurred_at`을 `Asia/Seoul`로 표시한다. `life_day.date`와 파일명은 저장 컨테이너 날짜이므로 실제 사건 날짜로 말하지 않는다.
- `occurred_at`이 null이면 실제 복용 시각과 날짜가 기록되지 않았다고 말하고 `recorded_at`이나 생활일 날짜로 보완하지 않는다.
- `skipped`는 복용·도포한 사실로 말하지 않고 건너뛴 기록으로 구분한다. 같은 약의 반복 이벤트는 모두 보존한다.
- 조회 중에는 medication add/take, remove, update/correct, day note와 audit mutation을 실행하지 않는다. history 명령은 파일, index, brief, Git을 쓰지 않는다.
- 사용자가 “오늘 한 거 맞잖아”처럼 반박하면 먼저 structured medication event를 다시 조회하고 설명을 정정한다. 반박만으로 새 기록·삭제·정정을 실행하지 않는다.
- 약 삭제·정정은 명시적인 의도가 있어야 한다. 후보가 여러 개이거나 하나로 식별되지 않으면 약 이름, action, 실제 발생 날짜·시각, 생활일 날짜, dose/note 요약을 보여주고 확인받는다. 삭제 뒤에는 history를 다시 조회해 남은 기록을 보고한다.
- reply timestamp는 메시지 전송 시각을 알려주는 보조 정보다. `occurred_at`과 자동으로 같게 만들거나 덮어쓰지 않는다. 둘이 충돌하면 충돌 사실만 설명하고 정정 여부를 확인한다.

## 취침

1. 사용자가 말한 취침 시각을 기록한다.
2. `neoctl day sleep --at <ISO_DATETIME>`를 실행한다.
3. 취침은 생활일을 닫지 않는다.

명시적인 취침 보고는 runtime에서 먼저 기록한다. 취침 전 확인 질문은 이 mutation을 지연·취소하는 선행 조건이 아니며, `sleep_at` 기록 성공을 먼저 알린 뒤 후속 질문으로 진행한다. 기록이 실패하면 성공했다고 말하지 않고 실패 상태를 알린다. "졸리다", "잘까?"처럼 취침 의도가 확정되지 않은 말은 자동 기록하지 않는다.

### 취침 전 마무리 — 기록 확인

취침 보고를 받으면, 미기록 항목을 선제적으로 확인한다.

**최우선 — 약 복용 확인:**

4. `neoctl day med status`를 실행하고 summary.missing을 확인한다.
   - 사용자가 별도로 설정한 복약 알림 정책이 있을 때만 미복용 항목을 확인한다.
   - 먹었으면 `neoctl day med take --name 약이름` (같은 약 반복 복용 가능)
   - 건너뛰었으면 `neoctl day med skip --name 약이름 --reason "..."`
   - 아직 안 먹었으면 먹고 오라고 알려준다.
   - 같은 약 taken 이벤트가 하나 이상 있으면 복용 완료로 간주한다.

**그 다음 — 일반 기록 확인:**

5. 오늘 식사 기록이 비어 있으면 "오늘 뭐 먹었어?" 하고 물어본다.
   - 다 말하면 `neoctl day meal add`로 하나씩 기록한다.
6. mood가 기록되지 않았으면 "오늘 기분 어땠어?" 하고 물어본다.
   - 답변을 받으면 `neoctl day mood set`으로 기록한다.
7. 진행 중인 외출이 아직 안 돌아온 상태면 확인한다.
   - "아까 외출한 데서 돌아왔어?" — 돌아왔으면 `neoctl day outing back`
8. 오늘 todolist 항목 중 미완료가 있으면 확인한다.
   - "이거 오늘 못 했네. 내일로 넘길까?" → 넘기면 `neoctl pending add`로 이동
9. 오늘 한 일 중 day-independent하게 남겨둘 게 있는지 확인한다.
   - 있으면 `neoctl pending add`
10. medical 업데이트가 필요하면 묻는다.
   - 병원 기록 요청이 있으면 `neoctl medical add` 또는 `neoctl medical record MEDICAL_ID`를 사용한다.

11. 다음 생활일을 미리 만들지 않는다.
12. 다음 `기상`에서 기존 생활일이 닫힌다.

다음 기상 뒤에 이전 생활일의 미확인 항목을 확인하면 당시 생활일 날짜를 먼저 식별하고 `neoctl day --date YYYY-MM-DD ...`로 그 컨테이너에 소급 기록한다. 새 생활일을 기본 대상으로 삼아 잘못 기록하지 않는다.

## 같은 기상일 파일이 이미 있는 경우

`neoctl day wake`가 기존 날짜 파일 때문에 실패하면 새 생활일을 억지로 만들지 않는다.

- 기상 시각 정정인지 확인한다.
- 기존 생활일을 이어 쓰려는 것인지 확인한다.
- 파일 복제나 UUID suffix를 만들지 않는다.

## 작업 형태

내부 값은 다음과 같다.

```text
1   프로젝트 작업을 주된 일정으로 삼는 날
0.5 외부 일정과 프로젝트 작업을 병행하는 날
0   프로젝트 작업을 계획하지 않는 날
```

실제 작업시간과 환산하지 않는다. 당일 `planned_allocation` 합계는 `workday_capacity`를 넘지 않는다.

## 기상 시각 정정

기존 생활일의 기상 시각이 처음부터 잘못 기록된 경우에만 정정한다. 날짜·timezone·수면 순서를 검증하고 다른 note, todo, meal과 UUID는 유지한다.

    neoctl day correct --date YYYY-MM-DD --field wake_at --value ISO_DATETIME --reason "정정 사유"

일반적인 현재 상태 변경과 과거 사실 정정은 구분한다. 정정 전에는 대상과 사유를 확인하고, 실패하면 workspace 파일을 쓰지 않는다.

## 생활일 무효화

잘못 만들어진 생활일은 물리적으로 삭제하지 않는다.

1. 무효화할 이유를 사용자에게 확인한다.
2. 승인 후 `neoctl day void --approve`를 실행한다.
3. 과거 생활일 수정은 변경 대상과 이유를 먼저 보여준다.
